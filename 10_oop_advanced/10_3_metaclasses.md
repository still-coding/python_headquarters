# 🧬 Метаклассы
```toc
minLevel: 2
```

> ⚠️ Метаклассы — самая продвинутая тема этого раздела. В повседневной разработке они почти не нужны. Читай этот файл, если хочется понимать внутреннее устройство Python или разбирать код фреймворков. Для большинства задач достаточно `__init_subclass__` — к нему мы придём в конце.

## Главная идея: классы — это тоже объекты

В Python **всё является объектом**, включая сами классы. Класс — это объект, который умеет создавать другие объекты.

```python
class User:
    pass

user = User()

# user — объект класса User
# User — объект класса ???
print(type(user))      # <class '__main__.User'>
print(type(User))      # <class 'type'>
```

Класс `User` — это объект класса `type`. То есть `type` — это «класс классов».

**Метакласс** — это класс класса. Он управляет тем, как создаются классы, подобно тому как класс управляет тем, как создаются его экземпляры.

| Уровень | Пример | Кто управляет созданием |
|---|---|---|
| Экземпляр | `user = User()` | класс `User` |
| Класс | `class User: ...` | метакласс `type` |

## `type` как функция создания классов

У `type` две роли:

1. **Узнать класс объекта** — `type(obj)`
2. **Создать класс динамически** — `type(name, bases, namespace)`

Обычное объявление класса:

```python
class User:
    species = "Homo sapiens"

    def greet(self):
        return f"Привет, я {self.name}"
```

Полностью эквивалентно вызову `type`:

```python
User = type(
    "User",                                    # имя класса
    (),                                        # базовые классы
    {
        "species": "Homo sapiens",             # атрибуты
        "greet": lambda self: f"Привет, я {self.name}",
    },
)
```

Оба варианта создают один и тот же класс `User`. Обычный синтаксис `class ...` — это просто удобная обёртка над `type(...)`.

## Создание своего метакласса

Метакласс — это класс, унаследованный от `type`. Чтобы указать Python, что при создании класса нужно использовать его, передают `metaclass=`:

```python
class Meta(type):
    def __new__(mcs, name, bases, namespace):
        print(f"Создаётся класс {name}")
        return super().__new__(mcs, name, bases, namespace)


class MyClass(metaclass=Meta):
    x = 10

# Сразу при объявлении выведется:
# Создаётся класс MyClass
```

Что произошло:

- `Meta.__new__` вызвался в момент объявления `MyClass`
- `mcs` — сам метакласс (аналог `cls` у обычных методов)
- `name` — строка `"MyClass"`
- `bases` — кортеж базовых классов
- `namespace` — словарь атрибутов тела класса

### Хуки метакласса

| Метод | Когда вызывается |
|---|---|
| `__new__(mcs, name, bases, namespace)` | При создании самого класса. Возвращает новый класс. |
| `__init__(cls, name, bases, namespace)` | После создания, для инициализации класса. |
| `__call__(cls, *args, **kwargs)` | При вызове класса: `MyClass()` |

Обычно достаточно `__new__` или `__init__` — разница нужна, только если ты планируешь подменить сам создаваемый объект-класс.

## Пример: автоматическая регистрация классов

Классическая задача — реестр плагинов, в который автоматически попадает каждый новый наследник.

```python
class PluginMeta(type):
    registry = {}

    def __new__(mcs, name, bases, namespace):
        cls = super().__new__(mcs, name, bases, namespace)
        if bases:    # не регистрируем сам базовый класс
            mcs.registry[name] = cls
        return cls


class Plugin(metaclass=PluginMeta):
    """Базовый класс плагина."""


class CSVExporter(Plugin):
    def export(self, data): ...


class JSONExporter(Plugin):
    def export(self, data): ...


print(PluginMeta.registry)
# {'CSVExporter': <class 'CSVExporter'>, 'JSONExporter': <class 'JSONExporter'>}
```

Каждый новый класс-плагин регистрируется сам — без декораторов, без ручных вызовов. Это то, что делает Django с моделями, SQLAlchemy с `Base`, многие системы плагинов.

## Пример: валидация структуры класса

Метакласс может проверить, что наследники соблюдают определённые правила:

```python
class RequireFieldsMeta(type):
    def __new__(mcs, name, bases, namespace):
        if bases:    # проверяем только наследников
            required = {"name", "price"}
            missing = required - namespace.keys()
            if missing:
                raise TypeError(
                    f"{name} должен определить атрибуты: {missing}"
                )
        return super().__new__(mcs, name, bases, namespace)


class Product(metaclass=RequireFieldsMeta):
    pass


class Book(Product):
    name = "Python для всех"
    price = 500
# ✅ ОК


class BrokenItem(Product):
    name = "Что-то"
    # нет price → TypeError: BrokenItem должен определить атрибуты: {'price'}
```

Ошибка вылетает в момент **объявления** класса, а не при попытке его использовать. Это ценно: проблема обнаруживается раньше.

## Наследование метаклассов

Если класс уже имеет метакласс, то его наследники автоматически используют тот же метакласс:

```python
class Meta(type):
    def __new__(mcs, name, bases, namespace):
        print(f"Создаётся {name}")
        return super().__new__(mcs, name, bases, namespace)


class Base(metaclass=Meta):
    pass


class Child(Base):     # Meta используется автоматически
    pass

# Выведет:
# Создаётся Base
# Создаётся Child
```

Это одновременно и плюс (удобно), и источник конфликтов: если пытаешься унаследоваться от двух классов с разными метаклассами, Python выбросит ошибку. Поэтому в одной иерархии обычно один метакласс.

## `__init_subclass__` — простая альтернатива

Большинство реальных задач, которые раньше решались метаклассами, в современном Python можно решить через **`__init_subclass__`** — специальный метод, который вызывается при создании **наследника** класса.

```python
class Plugin:
    registry = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        Plugin.registry[cls.__name__] = cls


class CSVExporter(Plugin):
    pass


class JSONExporter(Plugin):
    pass


print(Plugin.registry)
# {'CSVExporter': ..., 'JSONExporter': ...}
```

Сравни с примером с `PluginMeta` выше — результат тот же, кода меньше, метакласс не нужен.

### Когда `__init_subclass__` хватает, а когда нет

**Хватает:**

- регистрация наследников в реестре
- проверка обязательных атрибутов у наследников
- модификация атрибутов созданного класса
- 99% задач, которые раньше требовали метаклассов

**Не хватает:**

- изменить поведение класса при его **вызове** (`MyClass()`) — для этого нужен `__call__` метакласса
- влиять на namespace **во время** построения тела класса (а не после) — есть `__prepare__` у метакласса
- делать что-то с самим базовым классом до его полного создания

Для подавляющего большинства случаев — **бери `__init_subclass__`**. К метаклассам прибегай, только когда нужно что-то, чего `__init_subclass__` сделать не может.

## Когда действительно использовать метаклассы

- 🟢 Пишешь ORM или фреймворк, где класс должен вести себя особым образом (Django models, SQLAlchemy)
- 🟢 Нужен контроль над процессом создания класса, который `__init_subclass__` не даёт
- 🟢 Переопределяешь `__call__`, чтобы изменить создание экземпляров (Singleton — хотя и для него есть проще способы)
- 🔴 Всё остальное — предпочти `__init_subclass__`, декоратор класса или дескрипторы

Есть известная цитата Тима Питерса (одного из ключевых разработчиков Python):

> Метаклассы — это более глубокая магия, чем 99% пользователей должны когда-либо задумываться. Если ты сомневаешься, нужны ли они тебе, — значит, не нужны. Люди, которым они действительно нужны, знают это точно и не нуждаются в объяснениях, зачем.

## 💡 Главное

```python
# Классы в Python — это объекты класса type
# Метакласс — это класс класса

# type(name, bases, namespace) — создаёт класс динамически

# Свой метакласс:
class Meta(type):
    def __new__(mcs, name, bases, namespace):
        cls = super().__new__(mcs, name, bases, namespace)
        # модификация/регистрация/валидация
        return cls


class MyClass(metaclass=Meta):
    pass


# Метакласс наследуется автоматически
# Два разных метакласса в одной иерархии → конфликт

# __init_subclass__ — современная альтернатива,
# покрывает почти все реальные сценарии:
class Base:
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        # действия над наследником


# Метаклассы оправданы, если:
# - пишешь ORM/фреймворк
# - нужен контроль над вызовом класса или namespace
# - __init_subclass__ не хватает

# Во всех остальных случаях — __init_subclass__ или декоратор класса.
```
