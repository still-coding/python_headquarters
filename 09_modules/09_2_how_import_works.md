# 🔍 Как работает импорт
```toc
minLevel: 2
```

## Зачем это знать?

Когда пишешь `import something`, кажется, что происходит магия. Но это не магия — это обычный, предсказуемый процесс, который можно пошагово объяснить. Понимание этого процесса избавляет от главных головных болей с модулями:

- `ModuleNotFoundError: No module named 'foo'` — хотя файл `foo.py` точно где-то есть
- Изменения в модуле «не видны» — хотя файл точно сохранён
- `ImportError: cannot import name 'X' from partially initialized module` — загадочная ошибка циклического импорта
- Непонятные папки `__pycache__`, которые везде появляются

## Что происходит при `import`

Когда Python видит `import my_module`, он делает три вещи:

1. **Ищет модуль** — просматривает список папок `sys.path`
2. **Запускает код модуля** — построчно, сверху вниз
3. **Сохраняет результат в кэше** `sys.modules`, чтобы не повторять первые два шага

> 💡 Ключевой момент: `import` — это не «подключение заголовков» как в C/C++. Это **запуск файла**. Всё, что модуль делает «на верхнем уровне» (объявления функций, классы, вычисления), выполняется прямо сейчас.

```python
# greetings.py
print("Модуль greetings загружается!")   # Выполнится при первом import

def hello(name):
    return f"Привет, {name}!"
```

```python
# main.py
import greetings        # Напечатает: Модуль greetings загружается!
import greetings        # Ничего не напечатает — модуль уже в кэше

print(greetings.hello("Иван"))
```

## Шаг 1: `sys.path` — где Python ищет модули

`sys.path` — это обычный список строк с путями к папкам. Python ищет модули в этих папках **по порядку** и берёт первый найденный.

```python
import sys

for path in sys.path:
    print(path)

# Примерный вывод:
# /home/user/my_project                    ← папка запускаемого скрипта
# /usr/lib/python3.13                      ← стандартная библиотека
# /usr/lib/python3.13/lib-dynload
# /home/user/.venv/lib/python3.13/site-packages   ← сторонние пакеты
```

### Что попадает в `sys.path`

1. **Папка запускаемого скрипта** (или текущая папка в интерактивном режиме)
2. Пути из переменной окружения `PYTHONPATH`
3. Стандартная библиотека Python
4. Папки `site-packages` — туда устанавливаются пакеты через `pip`

### Типичная ошибка: «скрипт не видит соседний модуль»

```
my_project/
├── main.py
├── utils.py
└── tools/
    └── helper.py
```

```python
# main.py
import utils            # ✅ работает — utils.py рядом с main.py

# tools/helper.py
import utils            # ❌ ModuleNotFoundError
                        # helper.py запускается из папки tools/
                        # а utils.py лежит на уровень выше
```

> 💡 Решение — пакеты и запуск через `python -m`. Об этом дальше: [[09_3_packages|Пакеты]] и [[09_5_dunder_name|`python -m`]].

## Шаг 2: Выполнение кода модуля

При первом `import` Python **выполняет файл целиком**, сверху вниз. Всё, что оказалось в этом процессе в глобальном пространстве имён модуля, становится его атрибутами.

```python
# config.py
print("Читаю конфиг...")          # 1. Выполнится при импорте

DEBUG = True                       # 2. Станет config.DEBUG
VERSION = "1.0"                    # 3. Станет config.VERSION

def get_settings():                # 4. Станет config.get_settings
    return {"debug": DEBUG}

_calculated = 2 + 2                # 5. Станет config._calculated
```

```python
# main.py
import config
# Сразу напечатает: Читаю конфиг...

print(config.DEBUG)          # True
print(config.get_settings()) # {'debug': True}
```

### Разница между `import` и `from ... import`

```python
# Вариант 1
import config
config.DEBUG        # Обращаемся через модуль

# Вариант 2
from config import DEBUG
DEBUG               # Прямой доступ к имени
```

В обоих случаях файл `config.py` **выполняется целиком**. Разница только в том, какие имена появятся в текущем пространстве имён.

## Шаг 3: `sys.modules` — кэш загруженных модулей

После первой загрузки модуль попадает в `sys.modules` — словарь вида `{"имя_модуля": объект_модуля}`. Все последующие `import` просто достают оттуда готовый объект.

```python
import sys
import math

print("math" in sys.modules)       # True
print(sys.modules["math"])          # <module 'math' ...>

# Проверим: повторный import не перезапускает модуль
import math                         # Ничего не происходит
import math                         # Ничего не происходит
```

### Практическое следствие: модуль — это синглтон

Если два разных файла импортируют `config.py`, они работают с **одним и тем же** объектом модуля.

```python
# module_a.py
import config
config.DEBUG = False      # Поменяли значение

# module_b.py
import config
print(config.DEBUG)       # False — видим изменение из module_a
```

> ⚠️ Это же и ловушка: глобальные переменные модуля становятся общими для всей программы. Обычно это не то, чего хочется.

### Как сбросить кэш и перезагрузить модуль

Изменил код модуля, но в запущенном интерпретаторе `import` показывает старое? Это из-за кэша.

```python
import importlib
import my_module

# Перезагрузить модуль после изменений
importlib.reload(my_module)
```

> 💡 Обычно это нужно только в интерактивных сессиях (Jupyter, REPL). В обычных скриптах каждый запуск стартует с чистым `sys.modules`.

## `__pycache__` и файлы `.pyc`

После первого импорта рядом с модулем появляется папка `__pycache__` с файлами вида `my_module.cpython-313.pyc`:

```
my_project/
├── main.py
├── utils.py
└── __pycache__/
    ├── utils.cpython-313.pyc
    └── main.cpython-313.pyc
```

`.pyc` — это **байткод**, промежуточное представление кода, которое Python исполняет напрямую. Компиляция `.py` → `.pyc` происходит один раз; при следующем импорте Python проверит, что `.py` не менялся, и возьмёт готовый байткод.

**Что с этим делать:**

- 🟢 Не трогать — Python сам управляет кэшем
- 🟢 Добавить `__pycache__/` в `.gitignore` — в репозитории ему не место
- 🔴 Не коммитить `.pyc` файлы

## Циклические импорты

Если модуль `a.py` импортирует `b.py`, а `b.py` импортирует `a.py` — получается цикл.

```python
# a.py
import b

def func_a():
    return b.func_b()
```

```python
# b.py
import a                    # ← в этот момент a.py ещё не доисполнен!

def func_b():
    return a.func_a()
```

Что происходит при `import a`:

1. Python начинает выполнять `a.py`, кладёт **пустой** объект модуля `a` в `sys.modules`
2. Встречает `import b` → начинает выполнять `b.py`
3. В `b.py` встречает `import a` → находит `a` в `sys.modules` (он уже там, хоть и пустой)
4. `b.py` доисполняется, возвращает управление в `a.py`
5. `a.py` доисполняется

Если в `b.py` была строка `from a import func_a`, то на шаге 3 она **упадёт**: `func_a` ещё не определена, `a.py` не дошёл до её объявления.

**Как избежать:**

- 🟢 Пересмотреть архитектуру — обычно циклический импорт означает, что общую логику надо вынести в третий модуль
- 🟢 Импортировать внутри функции, а не на верхнем уровне (отложенный импорт)
- 🟢 Использовать `import a`, а не `from a import func_a`

## 💡 Главное

```python
# Что делает import:
# 1. Ищет модуль в sys.path
# 2. Выполняет код модуля сверху вниз (один раз)
# 3. Кладёт результат в sys.modules

# sys.path — список папок, где Python ищет модули
import sys
sys.path

# sys.modules — кэш загруженных модулей
"math" in sys.modules    # True после import math

# Повторный import не перезапускает модуль
import my_module         # Выполнится
import my_module         # Ничего не произойдёт

# Перезагрузка — только через importlib
import importlib
importlib.reload(my_module)

# __pycache__ — автоматический кэш байткода
# Не трогать, добавить в .gitignore

# Циклические импорты → сигнал о проблеме в архитектуре
```
